# Architecture

## Goals

- run a real model locally inside Nitro
- support a hosted OpenAI-compatible backend behind the same module API
- avoid Ollama and native runtime dependencies
- publish as a normal Nuxt module
- ship only JS/TS + WASM in the package

## Execution modes

The module resolves runtime behavior from `edgeAI.provider`:

- `local`: vendored Transformers.js + ONNX Runtime WASM inference inside Nitro
- `remote`: OpenAI-compatible API execution through `edgeAI.remote`
- `mock`: deterministic fake responses for tests and fixture validation

The legacy `runtime` option is still tolerated for backward compatibility, but `provider` is the primary configuration surface now.

## Runtime shape

`src/module.ts` wires the module into four Nitro routes under `edgeAI.routeBase`:

- `GET {routeBase}/health`
- `POST {routeBase}/pull`
- `POST {routeBase}/generate`
- `POST {routeBase}/chat/completions`

It also exposes the same route base through:

- `useEdgeAI()`
- `nuxtApp.$edgeAI`
- the exported `EdgeAI` client

## Local runtime path

The local provider lazily loads:

- vendored `transformers.web.js`
- vendored `onnxruntime-web` WASM backend

At runtime the server:

- installs a `file://` fetch shim so Node can read vendored WASM and optional local model assets
- injects the ONNX runtime through `globalThis[Symbol.for('onnxruntime')]`
- patches the Transformers.js device-selection path so `device: 'wasm'` remains available in Node
- caches the initialized pipeline in process memory, keyed by model source and dtype

## Why vendoring is used

Consumers should not need runtime npm dependencies such as `@huggingface/transformers` or `onnxruntime-web`.

So the module vendors:

- patched `transformers.web.js`
- `ort.wasm.min.js`
- `ort-wasm-simd-threaded.js`
- `ort-wasm-simd-threaded.wasm`

## Provider and fallback flow

- `pull()` warms the active provider and reports whether the model was loaded during that request
- `generate()` accepts either a plain `prompt` or chat-style `messages`
- in `local` mode, per-request `remote: true` forces remote execution when the remote provider is enabled
- in `local` mode, failed local warmup or generation can fall back to the remote provider when `edgeAI.remote.fallback` is enabled
- `chat/completions` is implemented by adapting the OpenAI-style request to the same generation engine

## Presets and config resolution

The local model config is now preset-driven:

- built-in presets are defined in `src/runtime/presets.ts`
- `edgeAI.preset` selects a named preset such as `distilgpt2`
- `edgeAI.model` applies partial overrides on top of the selected preset
- `edgeAI.presets` lets consumers register additional named presets
- public runtime metadata exposes `preset`, `presets`, `defaultModel`, and `remoteFallback`

## Packaging choices

- vendored runtime assets are generated by `pnpm vendor:runtime`
- the published package carries the exact JS/WASM files the runtime needs
- consumers do not need Rust, Python, C++, Ollama, or `llama.cpp`
- the earlier Rust proof-of-concept path has been removed; the maintained runtime path is JS/WASM only

## Why `mock` still exists

`mock` exists only for:

- test fixtures
- offline CI
- local module development without model downloads
